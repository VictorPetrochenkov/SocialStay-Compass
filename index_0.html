«Друзья рекомендуют»

Персонализированный подбор отелей на основе данных о клиенте

1. Цель проекта

Создать модель, которая по общедоступным признакам клиента предлагает один или несколько отелей, где он с высокой вероятностью захочет отдохнуть. При этом «друзья» (любые социальные связи) служат дополнительным усилителем качества рекомендации.

2. Задачи проекта

  • Сформировать и очистить датасет «клиент ↔ отель факт».
  • Выделить набор признаков, которые клиент может предоставить.
  • Разработать и обучить модель ранжирования/классификации.
  • Оценить метрики качества и бизнес-ценность.
  • Внедрить API-эндпоинт «дай мне подходящие отели».

3. Фабула (коротко)

Путешественник заходит в приложение. Он не вводит даты и не ищет «свободные номера». Система сразу видит его профиль и показывает 3-5 отелей, в которых, судя по «похожим» людям и отзывам друзей, ему почти наверняка понравится отдыхать. Пользователь может сразу забронировать или добавить в «мечты».

4. Подробное описание

4.1. Источники данных

Источник Что даёт Частота обновления
CRM / Личный кабинет демография, лояльность, история поездок ежедневно
Соцсети (опционально) список друзей, лайки/чек-ины по OAuth
OTA-бронирования hotel_id, даты, цена, канал в реальном времени
Каталог отелей координаты, звёзды, сервисы еженедельно

4.2. Список фичей, которые клиент может предоставить

Достаточно одного заполненного профиля или авторизации через соцсеть.

  • age — возраст (дискретно 18-24, 25-34, …)
  • gender — пол (M / F / other / unknown)
  • family_status — single, couple, family_with_kids, group
  • home_country — ISO-3166
  • home_city_population — tier 1/2/3 < 100 k/100 k-1 M/>1 M
  • loyalty_tier — none / silver / gold / platinum
  • avg_trip_length_nights — медиана по истории
  • price_preference_bucket — budget / mid / luxury
  • preferred_trip_purpose — leisure / business / mixed
  • preferred_climate — sea, mountains, city, ski, exotic
  • friends_hotel_ids — список отелей, где побывали ≥1 друг (опционально)
  • device_type — ios / android / web

4.3. Целевая переменная

hotel_id из каталога (≈ 25 000 уникальных отелей).
Задача формулируется как softmax-классификация или nearest-neighbor retrieval.

4.4. Как будет работать MVP

  1. Пользователь авторизуется (email или OAuth) → система собирает 12 фичей.
  2. Модель возвращает 5 hotel_id, отсортированных по вероятности.
  3. UI показывает карточки отелей с пометкой «2 друга останавливались здесь».
  4. Клик → переход к бронированию.

4.5. Оценка качества

Метрика Описание Целевое значение
Recall@5 доля пользователей, для которых «правильный» отель попал в топ-5 ≥ 0.35
NDCG@5 позиционно-взвешенная точность ранжирования ≥ 0.50
CTR → Бронь от клика по карточке до успешной оплаты +15 % к baseline

4.6. Дальнейший путь

  • Добавить контекстные фичи (дата вылета, длительность).
  • Учитывать насыщенность отеля, чтобы не рекомендовать sold-out.
  • Online-learning по фидбеку (лайк/дизлайк карточки).
  • Расширить соц-граф до 2-й степени друзей.
